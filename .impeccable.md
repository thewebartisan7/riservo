# Impeccable — Design Context for riservo.ch

> Read this at the start of any design task. It is the source of truth for aesthetic and UX decisions on this project.

---

## Design Context

### Users

riservo.ch serves **two distinct audiences** on the same platform. The aesthetic language is shared; the density and warmth dial change per surface.

**1. Business users — dashboard** (`/dashboard/*`)
- Swiss service-based professionals: hairdressers, therapists, coaches, consultants, independent practitioners
- Use the product **daily, for extended sessions** — calendar, bookings, customer directory, settings
- Context: desktop-first, often with second monitor, during their workday
- Jobs: glance at today's schedule, confirm/cancel a booking, fit in a walk-in, edit service hours, answer customer questions
- Expect **speed and precision** over decoration. Frustrated by clutter, charmed by considered details

**2. Customers — public booking + management** (`/{slug}`, `/bookings/{token}`, customer portal)
- End users of the businesses — broad age range, typically **35–60+**
- Mobile-first (embedded via iframe or popup on the business's own site is common)
- Context: **60-second transaction**, often in a moment of intent ("book a haircut now"), sometimes with divided attention
- Jobs: find a slot, book it, receive confirmation, occasionally cancel or reschedule
- Expect **effortless and warm**. Confused by density, reassured by clarity. Must succeed on small screens with large touch targets.

### Brand Personality

**Three words**: **calm**, **crafted**, **hospitable**.

- **Calm**: never alarming, never urgent without cause. Steady pacing, generous negative space, no anxiety-driving patterns (red dots, spinning badges, countdown timers unless truly necessary).
- **Crafted**: every detail looks chosen, not templated. Typography, spacing, and color decisions are intentional. The product should feel like it was made by people who care, for people who care.
- **Hospitable**: warm without being saccharine. Language is welcoming but direct. Error messages help, not scold. Empty states invite rather than apologize.

**Swiss-ness is expressed through *behavior*, not typographic pastiche.** No Helvetica-only pastiche, no oversized numerals, no explicit "Swiss poster" motif. Instead: grid discipline, restraint, nothing superfluous, microscopic attention to alignment and rhythm. The feeling of a well-kept workshop or a clean concierge desk — not a design history lecture.

**Voice**: confident and understated. Say less, mean more. The platform should **recede so the business is the hero** (this is the Cal.com model applied to bookings: the platform disappears, the shop owner's brand shows through).

### Aesthetic Direction

**Reference**: Cal.com — calm surface, platform gets out of the way, business (or user) content is foregrounded, interaction is predictable and fast.

**Anti-reference**: Fresha — too marketplace-y, too much platform ego, too colorful, too many promotional surfaces. Also avoid: Calendly's genericness, Square's dashboard sprawl, generic-SaaS gradient-hero patterns.

**Visual strategy**:

- **Shared design language, different density.** Same tokens, same type scale, same color system across surfaces — different applications:
  - **Public booking page**: airier, more generous whitespace, larger touch targets, slightly warmer palette application, brand-forward (the *business's* brand is what shines — riservo is a frame).
  - **Dashboard**: compact and information-dense, calendar grids and tables prioritized, tight vertical rhythm, functional over decorative.

- **Theme**: light and dark mode available for **both** surfaces. Public booking must work in dark mode because it's embedded into third-party sites via iframe/popup — those sites may be dark-themed, and the widget should adapt gracefully.

- **Brand accent**: **Burnt Honey** — `oklch(0.62 0.11 65)` light / `oklch(0.72 0.12 70)` dark. A muted, warm, Swiss-material accent (brass, honey, pine, leather). Used sparingly: primary CTA, focus ring, active state, selected slot. Everything else neutral or tinted-neutral. Never as decorative fill. Never as gradient.

- **Neutrals**: tinted very subtly toward the brand hue (chroma ≤ 0.01) to create subconscious cohesion between UI chrome and accent moments. Pure-gray neutrals read colder than this brand wants.

- **Typography**: the default Inter is serviceable but reflexive — explore a more distinctive sans pairing at the next polish pass. Never uppercase body. Maintain a clear modular scale with at least 1.25 steps.

- **No illustrations, no mascots, no icon-in-rounded-square treatments above every heading.** Iconography is functional (Lucide, currently in use), never decorative.

- **Shadow / elevation**: minimal. Use borders and background tint for hierarchy before reaching for shadow. When shadow is used, it should feel like a soft daylight shadow — not a drop shadow halo.

- **Motion**: respectful and purposeful. Ease-out-quart for entrances (180–220ms), crossfades for state. **No bounce, no elastic, no parallax.** `prefers-reduced-motion` must collapse all transitions to 0.01ms. Calendar interactions in particular: snappy, not springy.

- **Radius**: current `0.625rem` base is acceptable. Keep it consistent — no mixing rounded/square components on the same surface.

### Design Principles

These are the rules to fall back on when making any design decision:

1. **Recede so the business is the hero.** Both on the public page (the *business's* identity should shine through riservo's frame) and on the dashboard (the *data* is the content, not our chrome).
2. **Precision over flourish.** If a decorative element cannot justify itself functionally or emotionally, remove it. "Crafted" is earned by restraint, not accumulation.
3. **Density by surface.** Public pages breathe; dashboards pack. Same typographic system, different spacing applications.
4. **Warmth in materials, not ornament.** The palette, the paper-like whitespace, the tinted neutrals — these carry the hospitable feeling. Not emoji, not mascots, not exclamation marks.
5. **Calm defaults.** Don't alarm unless there's a real problem. Don't celebrate unless there's a real win. The product's emotional baseline is steady.
6. **Accessible on purpose, not by accident.** WCAG AA minimum across the board. Public booking flow: generous touch targets (min 44×44px on mobile), high contrast body text, never rely on color alone to communicate state. Respect `prefers-reduced-motion` and `prefers-color-scheme`.

## Operational Rules

- **Language**: all user-facing copy uses `__('key')` with English base strings. IT/DE/FR translations added pre-launch. Design for **German line length** — it often runs +30–40% longer than English. Avoid fixed-width button labels.
- **Components**: COSS UI primitives (Base UI under the hood) are the building blocks. Do not introduce a new component library. Compose from what exists in `resources/js/components/ui/`.
- **Tokens**: the full token set lives in `resources/css/app.css`. Extend that file when adding new semantic tokens; do not hardcode colors in components.
- **Dark mode parity**: every component must work in both themes. Never tune purely for light mode.
- **Mobile parity on public surfaces**: the booking flow must be verified on a 375px-wide viewport before any feature is called done.

## Implementation Hygiene

When proposing styling or refactoring, the code output must conform to these rules — no exceptions. Historical impeccable passes have drifted on every one of these, so the rules are enumerated explicitly.

1. **Tailwind utilities only — no custom CSS classes.** The only non-utility CSS allowed in `resources/css/app.css` is: (a) `@theme` tokens, (b) `:root` / `.dark` semantic token blocks, (c) `@keyframes` definitions, (d) the short `@layer base` that sets `html`/`body` defaults. If a utility bundle repeats 3+ times, extract it to a **COSS UI primitive or a riservo primitive component** (e.g. `<Display>` in `components/ui/display.tsx`) — not a `.custom-name` class.

2. **No inline `style={{...}}`.** Ever. Use Tailwind utilities including arbitrary values when a named token doesn't fit: `h-[72px]`, `max-w-[480px]`, `text-[clamp(1.75rem,1.3rem+1.6vw,2.5rem)]`, `[stroke-dasharray:60]`. Inline styles are untokenized, unthemed, and break under dark mode. **And if a spec value is *close* to a named token, prefer the token over an arbitrary value** — e.g. a 44px heading becomes `text-5xl` (48px), not `text-[44px]`; a 13.5px body becomes `text-sm` (14px), not `text-[13.5px]`. The design system is the source of truth; the original mock is not.

3. **No mouse event handlers for styling.** Don't use `onMouseOver`, `onMouseLeave`, `onFocus`, `onBlur`, `onMouseEnter` to toggle styles. Use Tailwind variants: `hover:`, `focus-visible:`, `aria-pressed:`, `aria-invalid:`, `data-loading:`, `data-pressed:`, `enabled:`, `disabled:`, `not-aria-pressed:`, `has-aria-invalid:`, `has-focus-visible:`. State-driven style lives in the className, never in JS.

4. **Prefer COSS UI primitives over raw HTML.** Use `<Button>`, `<Field>` + `<FieldLabel>` + `<Input>` / `<Textarea>` + `<FieldError>`, `<Avatar>` + `<AvatarImage>` + `<AvatarFallback>`, `<Skeleton>`, `<Card>` + `<CardPanel>`, `<Select>` (never native `<select>`), `<Dialog>` etc. Raw `<button>` / `<input>` / `<img>` is only acceptable when the primitive genuinely doesn't fit (calendar grid, honeypot field, decorative container). Document any such exception in `docs/UI-CUSTOMIZATIONS.md`.

5. **Never fork COSS UI primitives.** Do not modify files in `resources/js/components/ui/*.tsx` that came from upstream COSS. Override at the callsite via `className` + `cn()`. When you need a bundle that primitives don't offer (display typography, a project-specific composite), build a **new riservo-owned primitive** in `components/ui/` using the same Base UI `useRender` / `mergeProps` pattern — it lives next to COSS files but is clearly ours.

6. **Prefer named tokens over arbitrary values.** Use `text-sm` over `text-[14px]`, `tracking-widest` over `tracking-[0.12em]`, `leading-tight` over `leading-[1.05]`. Accept 0.5–2px drift as the cost of brand consistency. Keep arbitrary values only for genuine one-offs: display hero numerals, clamp-based fluid scales, layout constants, SVG stroke properties.

7. **Semantic theme tokens, not hex literals.** Reach for `text-foreground`, `bg-muted`, `border-border`, `text-primary`, `bg-honey-soft`, `text-destructive-foreground`. Never `text-[#d2a055]`, `bg-[#fff]`, `style={{color: 'red'}}`. If a color doesn't exist yet, add it to `:root` + `.dark` + `@theme inline` before using it.

8. **Animations declared in `@theme`, consumed via `animate-{name}` utilities.** To add one: `--animate-rise: rise 220ms cubic-bezier(0.2, 0.8, 0.2, 1) both;` in `@theme inline`, matching `@keyframes rise {}` block, use `animate-rise` at the callsite. Don't inline with `style={{animation: ...}}` and don't emit `.bespoke-class { animation: ... }` blocks. Naming is generic (not feature-scoped) because animations get reused.

9. **Translation interpolation uses `:placeholder` syntax.** Pass replacements as the second arg to `t()`: `t('with :name', { name: collaborator.name })`. Never `t('with {name}').replace('{name}', value)` or template literals that bypass the translator.

10. **Format helpers live in `resources/js/lib/*-format.ts`.** Date, price, duration, initials, and other shared formatters go in named modules. Extract on the second duplicated use; don't wait for the third.

11. **No surface-scoped CSS.** Never write `.booking-root { font-family: ... }` or `.dashboard-root { ... }`. If a rule is brand-wide, put it in `@layer base` on `html`. If it's treatment-specific but surface-agnostic, make it a primitive component (like `<Display>`). Dashboard and booking share one token system — they don't have separate "scopes".

12. **Button responsive size gotcha.** COSS UI `<Button>` size variants apply responsive heights like `h-11 sm:h-10`. A custom `className="h-12"` only wins below the `sm` breakpoint. When you want a fixed height across all breakpoints, pass both: `className="h-12 sm:h-12"`. Same caveat applies to any responsive utility you try to override.

13. **Validation error styling goes through the primitives.** `<Input>` invalid border lives on the **wrapper span**, not the inner input — override via `has-aria-invalid:border-primary`, not `aria-invalid:border-primary`. `<FieldError>` defaults to `text-destructive-foreground`; override to `text-primary` via className when the surface uses honey-for-invalid.

14. **Tooling signals are canonical, not vibes.** If a utility seems off, check `resources/css/app.css` `@theme` block for available tokens. If TypeScript complains, fix the code — don't suppress with `as any`. If `npm run dev` / `vite` emits a CSS error (e.g. "Cannot apply unknown utility class"), the browser is serving stale CSS; **always** check the dev-server error stream before chasing phantom HMR bugs.
